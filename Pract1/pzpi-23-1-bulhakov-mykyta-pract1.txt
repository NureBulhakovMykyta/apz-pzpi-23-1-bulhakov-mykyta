     3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Загальні відомості про шаблони (патерни) проєктування
     Шаблон (патерн) проєктування в інженерії програмного забезпечення -
це загальне, повторно використовуване рішення для типової проблеми, що виникає в певному контексті при проєктуванні ПЗ. Патерн не є жорсткою структурою чи фрагментом коду, який можна безпосередньо скопіювати в програму; це радше опис або концептуальний шаблон для вирішення певної категорії проблем, який можна адаптувати до різних ситуацій.
     Згідно з класифікацією Gang of Four (GoF), шаблони проєктування поділяються на три основні групи:
     * породжувальні (Creational) - відповідають за гнучке створення об'єктів;
     * структурні (Structural) - допомагають компонувати об'єкти та класи в більші структури;
     * поведінкові (Behavioral) - відповідають за ефективну комунікацію та розподіл обов'язків між об'єктами.
     Область застосування: Патерни широко застосовуються в об'єктно-орієнтованому проєктуванні. Вони допомагають писати код, який легко читати, підтримувати та розширювати, уникаючи дублювання, знижуючи зв'язність (low coupling) та гарантуючи дотримання принципів SOLID. Крім того, вони формують спільну термінологію для комунікації між розробниками.
     3.2 Опис обраного патерна
     Назва патерна: strategy (стратегія). 
     Тип шаблона (патерна): поведінковий (behavioral).
     Основне призначення: Патерн визначає сімейство алгоритмів, інкапсулює кожен із них в окремому класі та робить їхні об'єкти взаємозамінними. Це дозволяє алгоритмам змінюватися незалежно від клієнтського коду, який їх використовує, забезпечуючи можливість заміни поведінки системи "на льоту" (під час виконання).
     
     Передумови застосування: використання є доцільним у таких випадках:
     * коли система має багато схожих класів, які відрізняються лише способом виконання певної поведінки (алгоритмом);
     * коли потрібно використовувати різні варіації алгоритму всередині об'єкта та перемикатися між ними під час виконання програми;
     * коли клас містить масивні умовні оператори (великі ланцюжки if-else або switch) для вибору варіанту алгоритму;
     * коли потрібно ізолювати бізнес-логіку класу від деталей реалізації специфічних алгоритмів.
     Переваги:
     * можливість динамічно (під час виконання програми) перемикати алгоритми;
     * забезпечує виконання принципу відкритості/закритості (Open/Closed Principle): нові стратегії можна додавати без зміни існуючого коду;
     * заміна громіздкого успадкування на більш гнучку композицію;
     * сприяє ізоляції логіки, що зменшує дублювання коду та значно полегшує незалежне модульне тестування кожного алгоритму.
     Недоліки:
     * збільшення загальної кількості класів та інтерфейсів у системі, що може ускладнити проєкт, якщо алгоритмів небагато;
     * клієнтський код повинен бути обізнаний щодо відмінностей між конкретними стратегіями, щоб мати змогу обрати правильну;
     * можливі накладні витрати на продуктивність через створення додаткових об'єктів стратегій у пам'яті.
     3.3 Структура шаблона
     Опис структурних частин:
     1. strategy Interface (Інтерфейс Стратегії): загальний інтерфейс або абстрактний клас, який є спільним для всіх алгоритмів. Він визначає єдиний контракт (наприклад, метод execute() або calculate()), який Контекст використовує для виконання алгоритму;
     2. concrete Strategies (Конкретні Стратегії): набір класів, які імплементують Інтерфейс Стратегії. Кожен із них містить реалізацію конкретного варіанту алгоритму або поведінки;
     3. context (Контекст): клас, який делегує виконання певного завдання об'єкту стратегії замість того, щоб виконувати його самостійно. Він зберігає посилання на Інтерфейс Стратегії та має метод для заміни поточної стратегії "на льоту";
     4. client (Клієнт): компонент системи, який відповідає за вибір необхідної стратегії залежно від умов. Клієнт створює екземпляр конкретної стратегії та передає його Контексту.
     Опис взаємодії об'єктів: клієнт визначає поточну бізнес-потребу, створює відповідний об'єкт Конкретної Стратегії та передає його в Контекст (через конструктор або спеціальний setter-метод),. Контекст не несе відповідальності за вибір алгоритму. Коли настає час виконати завдання, Контекст викликає загальний метод виконання на Інтерфейсі Стратегії, який міститься в його пам'яті. Виклик автоматично делегується приєднаному об'єкту Конкретної Стратегії, виконуючи заданий алгоритм. Контекст і стратегія лишаються слабозв'язаними.
     

Рисунок 3.3.1 - UML діаграма шаблону Strategy
     
     UML-діаграми з поясненнями: На класичній UML-діаграмі патерну Стратегія (див. рис. 3.3.1) зображено клас Context, який має агрегаційний зв'язок (зберігає посилання) до інтерфейсу Strategy. Контекст містить поле -strategy: Strategy та метод делегування +doWork(data). Від інтерфейсу Strategy (з методом +execute(data)) реалізуються кілька класів ConcreteStrategyA, ConcreteStrategyB тощо. Ці класи містять власну реалізацію методу execute(data). Клієнтський код (Client) взаємодіє з Контекстом та Конкретними Стратегіями, передаючи обраний об'єкт-стратегію до Контексту перед початком виконання задачі.
     3.4 Реалізація шаблона (патерна)
     Опис реалізації мовою програмування: Python.
     Основні етапи:
     * створення базових інтерфейсів: спочатку визначається алгоритм, який має кілька варіацій (у нашому випадку - це логіка побудови маршруту в навігаторі). Для нього створюється загальний абстрактний базовий клас RouteStrategy (з використанням модуля abc), який оголошує абстрактний метод build_route(start, destination). Це гарантує, що всі майбутні алгоритми матимуть єдиний контракт і будуть взаємозамінними для Контексту;
     * реалізація конкретних класів: кожен окремий алгоритм виноситься у власний незалежний клас. Створюються класи CarStrategy, WalkingStrategy та PublicTransportStrategy, кожен з яких імплементує інтерфейс RouteStrategy та реалізує специфічну для свого виду транспорту логіку всередині методу build_route. Паралельно створюється клас Контексту - Navigator. На відміну від NavigatorLegacy, він більше не містить жорстко закодованих умовних конструкцій if-elif. Замість цього він приймає об'єкт стратегії через конструктор, зберігає його та за потреби оновлює через метод set_strategy(). Виклик методу build_route() у Контексті просто делегує виконання підключеному об'єкту стратегії;
     * демонстрація використання: у клієнтському коді (в блоці if __name__ == "__main__":) створюється екземпляр класу Navigator з початковою стратегією для автомобіля. Після побудови першого маршруту поведінка об'єкта динамічно змінюється прямо під час виконання програми (runtime) - клієнт передає Контексту новий об'єкт WalkingStrategy, що призводить до побудови пішохідного маршруту за тим самим запитом, взагалі не змінюючи внутрішній код навігатора.

57 class WalkingStrategy(RouteStrategy):

62 class PublicTransportStrategy(RouteStrategy):

65         return f"Маршрут транспортом: {start} -> {destination}"

67 # Контекст: використовує стратегії незалежно від їхньої внутрішньої логіки
68 class Navigator:
69     def __init__(self, strategy: RouteStrategy):
70         self._strategy = strategy
71
72     def set_strategy(self, strategy: RouteStrategy):
73         self._strategy = strategy
74
75     def build_route(self, start, destination):
76         # Делегування виконання алгоритму обраній стратегії
77         result = self._strategy.build_route(start, destination)
78         print(result)
79         return result
